# M3.4_AUDIT.md — Milestone 3.4: Code-Structural Metadata

_This document satisfies the M3.4 deliverable requirement: per-field
documentation of derivation path, confirmation that no additional
extraction is required, and a "considered and rejected" appendix._

---

## 1. Implementation Overview

M3.4 introduces exactly two new structural-metadata fields on
`programming_syntax` educational objects:

| Field | Added to | Emitted by |
|---|---|---|
| `code_language: Optional[str]` | `schemas/canonical_base.py` → `CanonicalObjectBase` | `modules/copyright_sanitizer._code_structural_metadata()` |
| `code_construct_types: List[str]` | `schemas/canonical_base.py` → `CanonicalObjectBase` | `modules/copyright_sanitizer._code_structural_metadata()` |

One new validator rule:

| Rule | Added to | Rule name |
|---|---|---|
| Enum-membership check for both fields | `modules/structural_validator.py` | `_check_code_vocab_membership` (rule 13) |

**No other file is changed.** `pipeline.py` is untouched — both fields
are derived inside `_code_structural_metadata()`, which `sanitize_educational_objects()`
already calls at the M3.2/M3.3 checkpoint.

## 2. Deterministic Derivation Path

```
Stage D extraction  →  educational_objects[].reusable_syntax  (raw code text)
        │
        ▼
copyright_sanitizer.sanitize_educational_objects()      [existing checkpoint]
    └── _code_structural_metadata(reusable_syntax)      [existing helper, extended]
            ├── code_line_count          (M3.2, unchanged)
            ├── has_code_content         (M3.2, unchanged)
            ├── code_language            (M3.4 NEW — keyword/regex over same text)
            └── code_construct_types     (M3.4 NEW — keyword/regex over same text)
        │
        ▼
structural_validator._check_code_vocab_membership()     [new rule 13]
    ├── code_language ∈ CODE_LANGUAGE_VOCAB     → ERROR if not
    └── code_construct_types[i] ∈ CODE_CONSTRUCT_VOCAB → WARNING if not
        │
        ▼
Compiler IR / production Chapter JSON  (new optional keys, schema 4.1.0)
```

**Input available at derivation time:** `reusable_syntax` — the full raw
code/pseudocode string produced by `ProgrammingSyntaxRecognizer` or
`PseudocodeRecognizer` and present on the object before
`sanitize_educational_objects()` discards it (or preserves it verbatim
when `PRESERVE_CODE_SNIPPETS_VERBATIM=1`).

**Classification method:** pure keyword/regex matching over the in-hand
string.  Detection patterns in `copyright_sanitizer.py`:

- `_CODE_LANGUAGE_PATTERNS` — ordered list of `(label, compiled_regex)`;
  first match wins; falls back to `"other"`.
- `_CODE_CONSTRUCT_PATTERNS` — list of `(label, compiled_regex)`;
  all matching labels included, result sorted for determinism.

## 3. Why No Additional Extraction Is Required

Both fields are computed from `reusable_syntax`, which is:

- **Already in memory** at the sanitization checkpoint (Stage D produced
  it; `_code_structural_metadata()` already reads it for `code_line_count`
  and `has_code_content`).
- **A pure string** — no file I/O, no PDF access, no VLM call is needed
  to read it again.
- **Discarded after `_code_structural_metadata()` returns** (unless
  `PRESERVE_CODE_SNIPPETS_VERBATIM=1`).

The two new fields are computed in the **same single call** that already
extracts `code_line_count` and `has_code_content`.  There is no second
pass, no new helper invocation, no new import.

Design constraints verified:

| Constraint | Status |
|---|---|
| No additional VLM calls | ✓ Pure regex |
| No new OCR pass | ✓ Input is already-extracted text |
| No PDF re-read | ✓ `reusable_syntax` is in memory |
| No new pipeline stage | ✓ Extension of existing `_code_structural_metadata` |
| No breaking schema change | ✓ Additive optional fields, `extra="allow"` |
| Deterministic | ✓ Regex over fixed input; `code_construct_types` sorted |
| No semantic reasoning | ✓ Keyword/pattern presence, not pedagogical intent |
| No educational interpretation | ✓ Syntax family classification only |
| No graph construction | ✓ No edges, no nodes |

## 4. Closed Vocabularies

### `code_language`

| Value | Detection signal |
|---|---|
| `python` | `def <name>(`, `import <name>`, `print(`, `elif`, trailing `:` |
| `c_cpp` | `#include`, `void main(`, `printf(`, trailing `;` |
| `java` | `public static void`, `public class`, `System.out.print` |
| `javascript` | `function <name>(`, `const <name> =`, `console.log(` |
| `sql` | `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `FROM`, `WHERE` (case-insensitive) |
| `pseudocode` | `begin`, `end`, `procedure`, `function`, `input`, `output`, `algorithm` |
| `other` | Fallback — no pattern matched |

### `code_construct_types`

| Value | Detection signal |
|---|---|
| `assignment` | `=` not preceded by `=`, `!`, `<`, `>` |
| `class_def` | `class <name>` |
| `comment` | `#`, `//`, `/*`, `*/` |
| `conditional` | `if`, `else`, `elif`, `switch`, `case` |
| `exception_handling` | `try`, `catch`, `except`, `finally`, `throw`, `raise` |
| `function_def` | `def <name>(`, `function <name>(`, `void <name>(`, `procedure <name>` |
| `import_statement` | `import <name>`, `include` |
| `loop` | `for`, `while`, `repeat`, `do` |
| `print_statement` | `print(`, `printf(`, `console.log(`, `System.out.print` |
| `return_statement` | `return` |

## 5. Considered and Rejected Fields

The following fields were proposed during the M3.4 architectural review
and explicitly rejected.  They are **permanently out of scope for M3.4
and Phase 1** and must not be reintroduced during implementation or
future milestones without a new architectural review.

### `related_object_ids: List[str]`

**Proposed purpose:** expose co-location/provenance links between
educational objects (same block, same equation cluster, nearby objects).

**Rejection reason:** Every co-location and provenance relationship this
field would encode is already deterministically reconstructable by Phase 2
from existing Phase 1 artifacts:

- `block_id` / `source_block_id` (Stage D + `kg_readiness._enrich_one`) —
  Phase 2 can `GROUP BY block_id` to get exact block co-location.
- `parent_block_id` / `derived_from` (`kg_readiness._enrich_one`) —
  identifies equation-cluster parentage.
- `nearby_educational_objects` (`kg_readiness._enrich_one`) — already
  written per object; covers sibling and window-based proximity.
- `cross_references` / `visual_references` (`kg_readiness._enrich_one`) —
  captures textual cross-block references.
- DST `SequenceEntry` — records every canonical object ID under each
  `HeadingNode` in reading order.

Additionally: Stage D produces exactly one EO per equation-cluster block
(it iterates the flat top-level block list; equation-line child blocks
are never passed to Stage D as separate entries). There is therefore never
more than one EO per cluster, making intra-cluster co-membership a
structurally empty scenario.

**Principle violated:** Convenience of pre-aggregation is not sufficient
justification. The field must only remain if it preserves deterministic
information that would otherwise be permanently lost after Phase 1. It
does not.

### `equation_role: str` (definition / derivation / substitution_instance / restated_law)

**Rejection reason:** Assigns a pedagogical function to an equation.
This is educational interpretation of the equation's role in the
curriculum — not a structural fact about its shape. Violates Design
Principle 2 (no educational interpretation).  Also redundant: `block_type`
and `educational_intent` are already present on the equation record.

### `pedagogical_pattern: str` (group_activity / individual_activity / safety_warning / …)

**Rejection reason:** Classifying an activity's pedagogical character
requires interpreting educational intent, not its structural shape.
`activity_type` (already produced by `content_blocks.py`) is the correct
existing structural signal. Violates Design Principle 2.

### `account_category: str` (personal / real / nominal)

**Rejection reason:** Although the lookup table itself is deterministic,
its content embeds accounting curriculum knowledge — it is a domain
interpretation of what the rule means pedagogically.  Determinism does
not exempt a lookup table from being "educational interpretation" if what
it encodes is curriculum classification.  Violates Design Principle 2.

### `visual_subtype: str` (labeled_parts / process_flow / data_plot / illustration)

**Rejection reason:** Classifying a diagram's visual sub-kind requires
understanding image content.  No deterministic, text-only derivation path
exists from `bbox`/`caption` alone.  Proper classification would require
a new VLM call (violates Design Principle 6) or a new image-analysis
pass (violates Design Principles 7–8).  Belongs to Phase 2 if anywhere.

### `step_dependency_order: List[int]`

**Rejection reason:** Determining which prior step a given step "consumes"
requires interpreting what each step does — semantic/content reasoning
about procedure meaning, not a structural fact about document shape.
Violates Design Principle 2.  Also largely redundant: sequential order is
already preserved by list position in `procedure_step_marker_types`.
